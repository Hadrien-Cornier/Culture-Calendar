## What's different from prior overnight runs

Prior runs (2026-04-18, 2026-04-19) committed to a feature branch and deferred pushing
until a human reviewed locally. That worked until the 2026-04-19 v12i promotion shipped
with five regressions nobody caught: the v12i internal design-note text leaked into the
masthead subtitle, the About section disappeared, Top Picks was top-N across all events
instead of top-of-the-week, review panels rendered as a single textContent paragraph
without the structured `parseReview()` sections, and the mobile filter sheet opened cut
off-viewport with no close mechanism.

This run's contract enforces **push-per-task with live-site verification**:

1. Each task implements its change inside the scope fence.
2. Local oracle: `pytest -q` must pass.
3. Commit with the Hadrien-Cornier identity → `git push origin main`.
4. Poll the changed file at the GitHub Pages URL until the deploy lands (typically
   17–37s; 5-min timeout → BLOCKED `deploy-timeout`).
5. Run `scripts/check_live_site.py` (built by T0.1, pyppeteer-driven) against a
   per-task assertion spec at `.overnight/check-T<ID>.json`.
6. If the live check fails: `git revert HEAD --no-edit` + push + wait + re-verify
   the site stabilised on the prior content, then BLOCKED. Never leave the live site
   in a broken state.

The result: every change is provably visible to a real Chromium against the deployed
site before the next task starts. No regression can ride along under cover of a
green local oracle. If a task ever lies about working, the very next push waits on
its own selector and would deploy-timeout instead of compounding the lie.

## Decisions worth recording

### Push-per-task with revert-on-failure beats branch-and-defer

The 2026-04-19 push-then-discover-404 episode is the direct motivation. By making
every task's "DONE" gated on a real Chromium hitting the live site, regressions
either fail fast (revert + BLOCKED) or never ship. The cost is ~30s of deploy-wait
per task (12 tasks × ~30s ≈ 6 min total this run); the benefit is provable
correctness against deployed state. For runs that touch the public site, this
discipline should be the default.

### Per-task `.overnight/check-T<ID>.json` specs over a single end-to-end suite

Each task owns the assertions that prove its own change works. T5.1 then composes
all five regression checks into a final gate. This means a per-task failure points
at exactly one task's change (cheap to revert, cheap to re-attempt) and the final
gate catches subtle interactions (e.g. T3.2's `:last-child` rule that broke
paragraph spacing was caught by the per-task check, not by T5.1).

### System Chrome over bundled pyppeteer Chromium on Apple Silicon

T0.1's first version used pyppeteer's bundled Chromium 117 — it crashed on launch
because the bundled binary is x64 and runs under Rosetta on this M-series host.
`2a5bab4` swapped to `executablePath: /Applications/Google Chrome.app/.../Google
Chrome` so the verifier launches the native arm64 Chrome. If this runner is ever
moved to an x64 host (CI, Linux), revert that path; on Apple Silicon, leave it.

### Top Picks "of the week" = 7d ahead, not Mon–Sun

T2.1 uses a rolling 7-day window from `Date.now()`, not "this calendar week".
Rationale: a user opening the site on Sunday evening should still see picks for
the week ahead, not have the list reset to empty because Mon–Sun has already
ended. This matches how the rest of the site treats "upcoming".
