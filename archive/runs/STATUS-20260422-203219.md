## Post-halt recovery (T5.3 unblock)

T5.3 (the strict 6/6 LLM gate) BLOCKED on first runner invocation: the LLM council returned `search-user: FAIL` twice in a row. Diagnosis was correct — the persona spec at `.overnight/personas/search-user.json` double-typed "Para" (once via a `js_truthy` assert, once via `pre_screenshot_actions`), producing "ParaPara" at screenshot time while ground truth was captured at the intermediate "Para" state. The LLM correctly flagged the contradiction. Per `CLAUDE.md`'s explicit T5.3 rule ("no retry on LLM FAIL — manual triage"), the runner halted.

Manual unblock (commit `32579e8`) fixed three stacked issues surfaced during triage:

1. **search-user double-type** — dropped the redundant `pre_screenshot_actions` block; the assert already drives the desired state.
2. **logistics-user ground-truth gap** — added `.event-venue-address` to the persona's asserts so ground truth reports it; expanded the system prompt to name expected DOM nodes by class. Fix-the-view, not cross-fingers.
3. **groupEvents field-drop regression** — `docs/script.js:groupEvents()` built a whitelisted output dict that dropped `venue_display_name` + `venue_address`. Data plumbing in `update_website_data.py` was correct (commit `b7b36cb`) but `docs/data.json` was stale (last regenerated 2026-04-19, commit `c45fdfd`) AND the client-side aggregator didn't carry the fields through. Widened the whitelist + backfilled stale data via new `scripts/backfill_venue_metadata.py`.

Added `tests/test_persona_spec_consistency.py` as regression guard against a persona spec driving the same input selector from both assert and pre-action paths.

Council re-run after unblock: **6/6 structural PASS + 6/6 LLM PASS**, zero `Verdict.*FAIL` strings in `docs/PERSONAS.md`.
