## Decisions worth recording

### Buttondown stub fallback over hard-failure (T2.2)
The signup form posts to `window.CC_CONFIG.buttondown_endpoint` if set, otherwise renders a "Coming soon — drop us a note in the meantime" message and quietly fires the `cc_subscribe_email` event with a `stub: true` property. Reason: deploys must not block on having the Buttondown account configured. Trade-off: a user who submits when the endpoint is empty gets a friendly message but their email is not captured. Acceptable because the stub state is transient (must be configured before announcing the list).

### Per-platform Plausible events instead of one `cc_share` with a property (T1.1, T5.1)
Each platform fires its own custom event (`cc_share_twitter`, `cc_share_bluesky`, …) rather than a single `cc_share` event with a `platform: "twitter"` property. Reason: Plausible's free tier exposes custom events as first-class metrics in the dashboard, but requires goals to be configured per event name. Per-platform names mean each shows up as its own goal-conversion bar without needing to filter by property. Trade-off: 9 goals to configure manually instead of 1.

### `_safe_slug` extracted to `scripts/_slug_util.py` (T0.1)
Two callers (`build_event_shells.py`, `build_rss_feed.py`) already used identical 5-line slugify functions; v2 added per-event JSON (T3.3) which would have made it three. Extracted to a shared module before the duplication grew. Trade-off: one more file in `scripts/`, but eliminates a subtle bug class where the slug for an event shell stops matching the slug for its `.json` mirror.

### `docs/config.json` is the boundary (T0.3, T2.1)
Client-side code can't read `config/master_config.yaml`. Rather than embedding the Buttondown endpoint in `index.html` at build time, `scripts/build_config_json.py` emits a small `docs/config.json` and `script.js` `fetch()`s it on init into `window.CC_CONFIG`. Trade-off: one extra HTTP request on page load (cached, ~200 bytes); upside: the same config loader can grow new keys without touching the HTML template.

### AI-crawler allowlist is permissive by default (T3.4)
`robots.txt` explicitly allows GPTBot, ClaudeBot, PerplexityBot, CCBot, Google-Extended, Meta-ExternalAgent, Amazonbot. Reason: the entire site is public cultural-event listings — no privacy concern in being crawled. Allowlisting makes the policy explicit (vs. the previous behavior of "default-allow because no `Disallow` line exists"). The `[persona-gate]` tag on this commit was honored because crawler policy is a deliberate IA decision worth surfacing.

### Per-event canonical JSON mirrors the JSON-LD (T3.3)
229 files at `docs/events/<slug>.json`, each a structured-data view of the same event the JSON-LD `<script>` block carries. Lets agent crawlers fetch a clean JSON without parsing HTML. Trade-off: 229 small files in `docs/events/` (~50KB total); upside: the API surface is now scrapable without HTML parsing or running JavaScript.

### LLM persona FAIL verdicts are mostly pre-existing (T6.3)
The full LLM council returned FAIL for 5 of 6 personas, but all 6 passed structurally (every asserted selector resolved). The qualitative critiques (mobile 375px overflow, logistics-user wanting venue addresses inline, review-reader wanting better section formatting) are largely the same findings the v1 council surfaced — they describe gaps in the existing design, not regressions introduced by this run. The `[no-council]` tag on T6.3 is in the queue notes precisely because that task IS the council run; gating the council against itself would be circular.

## Known gaps + follow-ups

None block the run; candidates for a future session.

1. **Buttondown endpoint placeholder until configured.** Until the human runs the manual step above, the signup form is a stub. Consider adding a small "Configure Buttondown" reminder to the deploy script's stdout when `buttondown_endpoint` is empty.
2. **`/api/*.json` is unversioned.** Today, agents that depend on the JSON shape will break silently if a future run changes a field. Consider an `X-API-Version` header or a `version: "1"` key in each payload, plus a CHANGELOG section dedicated to API breaks.
3. **`llms-full.txt` is regenerated wholesale on each build.** It dumps every event review. As the catalog grows, this file will inflate. A future task could split it into `llms-events.txt` + `llms-venues.txt` + `llms-about.txt` and have `llms.txt` link to the parts.
4. **`ai-agent.json` doesn't advertise rate limits or auth requirements.** The manifest currently lists endpoints only. If we ever rate-limit, we'll need to publish that metadata.
5. **Weekly digest archive is single-week.** `docs/archive.html` only lists `weekly/2026-W17.html` because that's all that exists on disk. A `--backfill` mode for `build_weekly_digest.py` would let the archive be browsable.
6. **Persona-LLM critiques worth triaging.** From `docs/PERSONAS.md`: mobile-user (375px overflow on event titles), logistics-user (venue address not visible without click), review-reader (review-panel formatting). All pre-existing; none introduced by this run, but worth a future session.
7. **Share menu CSS hardening for ≤320px viewports.** T1.3 polished for 375px (default mobile breakpoint). The 9-button popover may still wrap awkwardly on iPhone SE / Galaxy Fold front display.
8. **No PR template enforcing the manual steps above.** A `pull_request_template.md` checklist with "Buttondown endpoint configured", "Plausible goals added", "Sitemap submitted to GSC" would prevent these slipping through review.
