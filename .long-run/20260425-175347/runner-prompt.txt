# Runner prompt template

This is the prompt the bash runner passes to each `claude -p` invocation, with `{{...}}` placeholders interpolated per task before the call.

The bash runner owns the queue. Each `claude -p` call executes exactly one task and exits. The prompt does not need to fight premature stopping — that's the runner's job.

**IMPORTANT for the runner**: the script's `render_prompt` function strips everything above the first standalone `---` line below before sending to `claude -p`. Keep that separator intact and put any human-readable preamble above it.

---

# Prompt body

You are executing **one queued task** in an unattended long run. The bash runner that invoked you owns the queue — do not try to advance past your assigned task, do not start other tasks, do not declare the run complete. Just do this one task well and exit.

**Your work will be reviewed.** Once you write a DONE result file, a council of reviewers will look at your commit diff, your CHANGELOG entry, the scope fence, and your test changes. They are looking for: files touched outside the task's scope, tests weakened or skipped to make the task pass, collateral damage outside scope, commit/CHANGELOG that misrepresents what changed. **The council requires unanimous agreement** — any single reviewer returning FAIL re-queues this task, regardless of finding severity. Do the work right the first time: keep the diff narrow, never weaken test assertions, write a commit message that accurately describes the diff.

**Pre-existing problems are in scope.** If the reviewers surface a defect that was already in the code before your task, you are still expected to fix it (inside your files-in-scope or the scope fence). The run's operating principle is that we fix problems when we find them — "it was already broken" is not a valid reason to leave a known defect in place after you have touched the area.

## Repo

Working directory: `{{REPO_PATH}}` (this is your CWD). This path is a **git worktree** — the main checkout of the repo lives at a different path and must not be touched; stay inside `{{REPO_PATH}}` for every file edit and git command.
Branch: `long-run/{{RUN_ID}}` (already checked out in this worktree — do not switch branches).
Read `CLAUDE.md` for the full run context, especially the **Hard constraints**, **Scope fence**, and **Working principles** sections. Honor them absolutely — the Working principles (think before coding, simplicity first, surgical changes, goal-driven execution) are what the council of reviewers is implicitly grading you on.

## Contract

1. Implement the change for this task only. Stay inside the listed files. If the task genuinely requires editing a file outside the listed scope, **stop and write a BLOCKED result file** (step 6) — do not silently expand scope.
2. Run the validation command:
   ```
   {{VALIDATE}}
   ```
   It must exit 0. If it fails, fix and retry once. If it still fails, **stop and write a BLOCKED result file** (step 6).
3. Run the full validation oracle from `CLAUDE.md` (test, lint, typecheck, build) — all must be green. If any oracle command fails, fix and retry once, then **write a BLOCKED result file** if still red.
4. Stage in-scope files and commit your changes. Use `git add <explicit files>` only; never `git add -A`. Commit message:
   ```
   {{COMMIT_TYPE}}(task-{{ID}}): {{TITLE}}
   ```
5. **Get the real commit SHA.** Immediately after the commit, capture it via `git rev-parse HEAD` (the Bash tool is available to you). Do NOT guess, predict, or fabricate a SHA — the council's morning-reviewer will diff the CHANGELOG claim against `git show` and flag any mismatch as a finding.
6. Append one entry to `CHANGELOG.md` inside this run's `<!-- BEGIN LONG-RUN: {{RUN_ID}} -->` fence, then commit the CHANGELOG as a separate commit `chore(task-{{ID}}): changelog`. The SHA in this entry refers to the **previous** commit (the code commit from step 4):
   ```
   ### task-{{ID}} — DONE — {{ISO_NOW}}
   - commit: <exact SHA from step 5, full or short — never guess>
   - files: <comma-separated list of files you actually touched, matching the diff>
   - summary: <ONE-SENTENCE description of what changed and why — name the function/symbol/value, not just the file. e.g. "Added isFutureOrToday() helper; main listings now filter past events using the same date-parse pattern as Top Picks". The morning-reviewer reviewer treats a missing or vague summary as a high-severity finding and re-queues the task.>
   - validation: green
   ```
   Two small commits per task (code + changelog) is fine — the council accepts this. A single amended commit would also work but requires `git commit --amend` which is allowed for your own HEAD only.

   Project-specific note for Culture Calendar: this repo has Python tests under `tests/` but **no JavaScript test framework**. Behavior changes inside `docs/script.js` and `docs/styles.css` are validated via (a) the task's `validate` grep check, (b) `tests/test_check_live_site.py` (pyppeteer DOM-assertion harness), and (c) the persona council's structural checks. If a reviewer flags `NO_TESTS_FOR_BEHAVIOR_CHANGE` for a JS-only change, **rebut it** — the project's design choice is no JS test framework. Cite this paragraph as evidence in `.long-run/{{RUN_ID}}/rebuttals/task-{{ID}}-test-integrity-critic.md`.
6. **As your absolute last action**, write a result file the runner will read. This is how the runner knows your disposition — `claude -p`'s exit code is not enough. Use the `Write` tool to create `.long-run/{{RUN_ID}}/task-result.json` (overwriting any prior content) with one of these exact shapes:

   On success:
   ```json
   {"status": "DONE", "task_id": "task-{{ID}}", "commit": "<sha>", "files": ["..."]}
   ```

   On blocked:
   ```json
   {"status": "BLOCKED", "task_id": "task-{{ID}}", "reason": "<one short line — no newlines, no tabs>"}
   ```

   The runner removes this file before invoking you, so a missing file after you exit is treated as a crash and surfaces as `TASK_BLOCKED … no-result-file`. Do not skip step 6 under any circumstance — even if you crash, the runner can recover; if you silently succeed without writing the file, the run halts.

## If this is a retry after a council rejection

Check the env var `LONG_RUN_TASK_RETRY`. If it is `0` (or unset), this is a fresh attempt — skip this section. If it is `>= 1`, the council rejected your prior attempt and you are now on rebuttal round `$LONG_RUN_TASK_RETRY` of 2. Your prior commit has been reverted by the runner, so HEAD is back to where you started.

1. Read `.long-run/{{RUN_ID}}/reviews/task-{{ID}}-verdict.json` for the aggregate blocking reason and the list of dissenting personas.
2. For each dissenting persona, read `.long-run/{{RUN_ID}}/reviews/task-{{ID}}-<persona>.json` in full — the `findings[]` array is where the detail is.
3. For each finding, pick one:
   - **Fix it.** If the finding is correct (whether you introduced the problem or it was pre-existing and you'd uncovered it), fix the code. Pre-existing problems are in scope — do not grandfather old defects.
   - **Rebut it.** If you genuinely believe the finding is wrong, use the Write tool to create `.long-run/{{RUN_ID}}/rebuttals/task-{{ID}}-<persona>.md` with a 1–3 paragraph explanation backed by concrete evidence: quote specific lines of the diff, output of tests, passages from repo docs/CLAUDE.md that contradict the finding. Don't just assert — show. The persona will re-read your rebuttal and must explicitly withdraw or hold.

You may fix some findings and rebut others — the two are not mutually exclusive, and addressing every finding (by either route) is required. A finding that is ignored without a fix or a rebuttal will come back on the next review.

After you either fix or rebut every finding, proceed through the normal Contract below (re-commit, re-update CHANGELOG, write the DONE result file). The runner will re-invoke the council; reviewers who see their rebuttal addressed will withdraw; reviewers who are unconvinced will hold. If any reviewer still holds after 2 rebuttal rounds, the task becomes terminally BLOCKED and the user decides whether to reattempt manually.

## Forbidden moves (reminder — also in CLAUDE.md)

- No `git push`, `git rebase`, `git reset --hard`, `git checkout main`.
- No `npm install` / `pip install` / `cargo add` / `go get` of new dependencies. If you think you need a new dependency, exit 2 with `needs-dep: <name>` on stderr.
- No edits outside the scope fence in CLAUDE.md.
- No interactive prompts — you have no human attached.

## How the runner reads your result

The runner ignores `claude -p`'s exit code for disposition (the CLI exits 0 even when the agent decides "I'm blocked" — the agent can't influence that). It reads `.long-run/{{RUN_ID}}/task-result.json` instead. So:

- File present, `"status": "DONE"` → `TASK_DONE` event.
- File present, `"status": "BLOCKED"` → `TASK_BLOCKED` event with your `reason`.
- File missing → `TASK_BLOCKED … no-result-file` (treated as crash; consecutive-blocker counter increments).
- `claude -p` exits non-zero → logged but interpretation still hinges on the file.

## Output discipline

Be terse. Don't narrate. Your stdout is captured into a per-task log file for debugging — but the only signal the runner acts on is `.long-run/{{RUN_ID}}/task-result.json`.
