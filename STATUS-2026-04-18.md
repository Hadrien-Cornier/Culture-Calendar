## Decisions worth recording

### Client-side-only TTS (T4.1)

We chose `window.speechSynthesis` over pre-generating MP3s with a paid API. Rationale:

- **Zero repo growth** — no binary audio files committed; the `docs/` folder stays lightweight for GitHub Pages.
- **Zero recurring cost** — no ElevenLabs / OpenAI TTS / Google Cloud TTS bill. Web Speech API is browser-native and free.
- **Zero new deps** — satisfies the overnight hard-constraint "no new Python deps, no new paid API deps".
- **Reasonable quality floor** — modern platform voices (Siri on iOS, Google on Android, native on desktop) are good enough for a weekend-culture listings site. Not studio quality, but accessible.
- **Trade-off accepted**: voice selection is opaque and varies by platform; no control over prosody or pronunciation of Austin-specific venue names.

Implementation notes for future maintainers:

- `voiceschanged` event is listened for because Chrome populates `getVoices()` asynchronously on first call.
- `speechSynthesis.cancel()` is always called before `speak()` to avoid queue buildup when users click rapidly.
- No polyfill — browsers without Web Speech get no button (feature-detection at render time).

## Open follow-ups

None of these are blockers for the run; all are candidates for a future overnight queue.

1. **Voice quality comparison across browsers** — no objective measurement exists yet. Ideas: record 30 s samples of the same description on each target browser (Chrome/Safari/Firefox desktop; iOS Safari/DuckDuckGo; Android Chrome/DuckDuckGo/Firefox), MOS-score them, and decide whether the quality gap is wide enough to justify a paid TTS for at least the "Picks" subset.
2. **Voice selection UI** — currently we let the browser pick the default voice. Some platforms expose 5–10 English voices. A small voice-picker dropdown (persisted in `localStorage`) would let users choose once.
3. **Pronunciation hints** — Austin venue names ("Alienated Majesty", "La Follia", "Hyperreal") sometimes get mangled by platform voices. SSML is not uniformly supported via the Web Speech API; a phonetic-substitution pass on the text before `speak()` is the pragmatic fix.
4. **Rate/pitch controls** — the API exposes `utterance.rate` and `utterance.pitch`. Worth exposing as sliders once voice selection lands.
5. **Resume / pause / seek** — today the button is play/stop only. `speechSynthesis.pause()` and `.resume()` are supported but inconsistently across mobile browsers; would need per-browser testing.
6. **iOS DuckDuckGo** — flagged in the run spec as a cross-browser target; worth a hands-on confirm since DDG's WKWebView-based rendering occasionally lags Safari in Web Speech behavior.
7. **Paramount scraper placeholder strategy** — T2.3 skips sparse-metadata events; consider a follow-up that backfills from the Austin.com or Do512 listings instead of dropping them silently.
