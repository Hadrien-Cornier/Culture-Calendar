## Decisions worth recording

### Feed granularity (T1.1 vs T1.2)
`calendar.ics` exports the full event universe; `top-picks.ics` only exports events with rating ≥ 8. This split lets a power user follow everything while casual users follow a curated stream. Both use the same base generator and stay inside `docs/` so GH Pages serves them without auth.

### OG card file shape (T2.2)
One SVG per *occurrence*, not per *event title* — because a repeated screening of the same film at different times needs its own share URL and thus its own card image. SVG (not PNG) was chosen to avoid a raster build step and keep the site cache-friendly; `docs/og/` currently holds ~1000 files, all static. If crawlers ever prefer PNG, a rasterise step can be grafted on without changing the generator API.

### Taste graph runs client-side only (T3.1–T3.4)
`localStorage.cc_taste` holds thumbs + saves; re-rank is a pure function executed on render. No server, no identity, no paid dep — matches the run's "no new paid API deps" constraint. Downside: signals don't sync across devices; upside: zero ongoing cost + aligns with the no-login product stance.

### Composer-of-the-week picked by event count (T8.1)
`build_composer_feature.py` picks the composer with the most upcoming concerts this week, then asks `LLMService` (existing Anthropic client) for a short essay. If the LLM abstains, the script writes a stub page with "Essay generation skipped" so the sitemap still resolves. Reason: weekly cadence means one flaky LLM call can't break the build.

### Per-person .ics lives inside `build_people_pages.py` (T8.3)
Rather than a new `build_people_feeds.py`, I extended the existing generator because the person → events mapping was already materialised there. One generator, two outputs. Trade-off: `build_people_pages.py` now carries both HTML-templating and iCal logic — acceptable under 300 lines, would need a split if it grows past 500.

### CHANGELOG SHA backfills (T8.2, T8.3, T9.1)
The runner writes the CHANGELOG entry in the same commit as the task output, but the CHANGELOG edit has to be staged before `git commit` — so the SHA in the entry is always one commit behind reality. For tasks where the human reads the SHA later (visible diffs), I added a follow-up `docs(task-<ID>): backfill commit SHA in CHANGELOG` commit. Three tasks tripped this (T8.2, T8.3, T9.1); the earlier ones self-resolved because the runner template used `<sha>` placeholder. The correct long-term fix is a post-commit runner hook that substitutes the real SHA into CHANGELOG and amends; noted as a follow-up below.

## Known gaps + follow-ups

None of these block the run; they are candidates for a future session.

1. **OG cards count growth** — ~1000 SVG files in `docs/og/`. Healthy for now but will grow monotonically. Add a prune step (drop cards for events older than 30 days) before this hits 5000.
2. **People-page selection threshold** — `build_people_pages.py` currently emits a page for anyone with ≥2 events. This produced a handful of pages for generic authorships ("Various Medieval Composers"). Consider a minimum event count of 3 or a name-quality heuristic (skip aggregate names like "Various…").
3. **Weekly digest is single-week** — only `docs/weekly/2026-W17.html` exists post-run because the generator keys on "the ISO week containing `today`". A retrospective backfill (`--backfill`) would let the digest archive be browsable.
4. **Plausible domain placeholder** — `data-domain="hadrien-cornier.github.io"` in the script tag is aspirational: the Plausible account needs to be created and the domain verified before events actually log. Until then the custom-event calls are silent no-ops (desired behavior — no error, just no stats). Flag for the morning: go create the Plausible account, then events flow.
5. **Audio brief voice selection is OS-dependent** — the brief uses the default `speechSynthesis` voice, which varies across OSes. On iOS the default Siri voice is usable; on Android Chrome the default can be robotic. A future task could enumerate voices and pick the highest-quality one that matches the `en-US` locale.
6. **Screenshot harness is single-URL** — `scripts/capture_screenshots.py --offline` currently snapshots index (+ mobile + How it works). A richer harness would also snapshot a venue page, a person page, and a weekly digest — would need ~5 more `--page` invocations.
7. **No visual-regression gate yet** — screenshots are artifacts, not assertions. A follow-up could diff the post-run PNGs against committed baselines and fail the gate on >5% pixel delta.
8. **Sitemap is static** — `build_sitemap.py` enumerates what exists on disk, not what's intended to exist. If a generator misses (e.g. people-pages generator crashes mid-run), the sitemap silently forgets those URLs. A loose check against feature-inventory is a reasonable watchdog.
9. **Runner SHA-in-CHANGELOG ordering** — as noted in "Decisions worth recording", three backfill commits this run. Fix: teach the runner to write CHANGELOG AFTER commit succeeds and either (a) use `git commit --amend` on the same task's commit or (b) let the follow-up docs commit be the canonical pattern (current behavior — acceptable, just noisier).
